Defining new effects

Non-pure side-effects in F* are modelled through computation types, also known as effects. We take a look at how to define and use new custom effects to represent state in an external system. The example we use is a set of valves controlling the flow of water into and out of a water tank.

Introduction

Suppose that we are writing software to control the flow into and out of a water tank. The tank has four valves, as shown in Figure 1. Each valve can be fully open, meaning water flows through it, or fully closed, which means that no water flows through it.

To control the valves, we are given the following functions, which we assume are defined and implemented in an external library:

  • open_all, which opens all valves.
  • close_all, which closes all valves.
  • open_valve, which takes an integer and opens the corresponding valve.
  • close_valve, which takes an integer and closes the corresponding valve.

The input valve has index 0, while the output valves have indices 1, 2, and 3.

Moreover, suppose that there are two constraints that arise from the physical environment:

  • Mechanical constraint: If the open_valve function is called to open a valve which is already open, the mechanical parts involved in opening the valve will be damaged. Similarly, using close_valve to close a valve which is already closed will also lead to mechanical damage. However, the open_all and close_all functions have no such restrictions; they are safe to call at any time.
  • Overflow constraint: The three output valves all have the same flow rate, but the input valve has a slightly larger flow rate. This means that if the input valve is open, then at least two output valves must be open to prevent the tank from overflowing.

Note that these constraints are of different kinds: The mechanical constraint is a constraint on state transitions, but does not say anything about the valid states themselves, while the overflow constraint limits the set of valid states.

There are multiple ways of writing code which breaks these constraints by mistake. We will define an effect and give these functions types which ensure that the constraints are enforced by the type checker.

Modeling the state of the valves

The state of the water tank system is completely determined if we know whether each of the four valves is open or closed. (We will not model the water level in the tank other than indirectly through the overflow constraint.) Conversely, every combination of open/closed states for the individual valves is a possible state for the entire system, even if some of the states will cause the tank to overflow. This means that the state of the system can be encoded as a list of four boolean values, where true means the valve is open and false means that the valve is closed. To make our code more readable we assign the name num_output_valves to the number of output valves, which in our example is 3. We encode this in F* as follows:

let num_output_valves = 3

let valve = x:int{x >= 0 /\ x <= num_output_valves}

type state = l:list bool{length l = 1 + num_output_valves}

Defining an effect

Having encoded the state of the system into a type, we now want to define an effect to use for typing the functions for controlling the water tank valves. We will define our effect in terms of the STATE_h effect from FStar.Pervasives, which will do most of the heavy lifting for us. Our effect definition looks like this:

new_effect TANK = STATE_h state

unfold let pure_lift (a:Type) (wp:pure_wp a) =
  fun (p: a -> state -> Type) (x: state) -> wp (fun y -> p y x)
sub_effect PURE ~> TANK = pure_lift

The first line defines a new effect, called TANK, which is the effect of functions which depend on or modify the state of our water tank system, which has state space state. The last line defines an embedding of the PURE effect into the effect TANK. The reason this is important is that we want to be able to call pure functions from TANK functions. The embedding is the function pure_lift. We explain how it works in the appendix, but for our current purposes it is fine to view this as a magic incantation.

To have an effect which is easier to work with, we define an effect which works with pre- and post-conditions, instead of the weakest precondition predicate transformers of the TANK effect.

effect Tank (a: Type)
            (pre: (state -> Type))
            (post: (state -> a -> state -> Type))
  =
  TANK a (fun p vs0 -> pre vs0 /\
            (forall x vs1 . pre vs0 /\ post vs0 x vs1 ==> p x vs1))

This allows us to write functions with types like f: a -> Tank b pre post. Here, pre is a precondition on the current state, which is required to hold before the function can be called, while post is a postcondition on the initial state, return value, and final state of executing the function, which will need to be proved to hold for the type checker to accept the function. What the above code block is saying is that to prove that some property p holds for the return value and final state of the function, it needs to be shown that the initial state vs0 satisfies the precondition pre, and that the property p follows from the precondition and postcondition of the function. Again, for the purposes of this post it is fine to not understand the details of this.

Using the effect

We are now ready to give types to the functions open_all, close_all, open_valve, and close_valve, showing how they modify the state of the water tank. The types we will give them describe the effect they have on the water tank system, but will not take the overflow constraint into account. For this reason, we will use the names open_all', close_all', open_valve', and close_valve' in this section, reserving the unprimed names for the final interface of the system.

We will assume that the actual implementations of the functions open_all', close_all', open_valve', and close_valve' are contained in an external library, and that we only need to write an interface file giving them F* types. For more details about this approach to external libraries, see Using OCaml libraries in F*.

Let us start with open_all' and close_all'. After open_all' has run, all valves are open. In other words, the final state vs satisfies forall i. (index vs i) = true. Similarly, the state vs after close_all' satisfies forall i. (index vs i) = true. Hence we can give the functions the following types to capture their effect on the state of the water tank.

val open_all':
    unit ->
      Tank unit
        (fun _ -> True)
        (fun _ _ vs -> forall i. (index vs i) = true)

val close_all':
    unit ->
      Tank unit
        (fun _ -> True)
        (fun _ _ vs -> forall i. (index vs i) = false)

The functions open_valve' and close_valve' are slightly more complicated. We know that after open_valve' i has been executed, the valve with index i is open. This can be expressed as index vs1 i = true where vs1 is the final state. Another important property of open_valve' is that it does not affect the state of any of the other valves, so that for all indices j apart from i, it holds that index vs0 j = index vs1 j, where vs0 is the initial state. Finally, we can take this opportunity to encode the mechanical constraint: It is only permissible to run open_valve' i when valve i is not already open. These properties are expressed in the following type.

val open_valve':
    i: valve ->
      Tank unit
        (fun vs -> index vs i = false)
        (fun vs0 _ vs1 -> index vs1 i = true /\
          (forall j. i = j \/ index vs0 j = index vs1 j))

Similarly, close_valve' i can only be called when valve i is open, and results in valve i being closed while all other valves are unchanged.

val close_valve':
    i: valve ->
      Tank unit
        (fun vs -> index vs i = true)
        (fun vs0 _ vs1 -> index vs1 i = false /\
          (forall j. i = j \/ index vs0 j = index vs1 j))

Some lemmas

In the next section we will need some lemmas, which are included here for completeness. To make the presentation less cluttered, we discuss them in this dedicated section. Feel free to skip this section if you are interested specifically in the code relating to state and effects.

We begin by proving that if all values of a nonempty list are a, then the number of elements equal to a is the length of the list:

val counting_lemma: l:list bool -> a:bool ->
            Lemma (requires length l > 0 /\ (forall i. index l i = a))
                  (ensures count a l = length l)
let rec counting_lemma l a = match l with
  | h :: [] -> assert (index l 0 = a)
  | h :: tl -> assert (forall (i:nat{i+1 < length l}). index l (i+1) = index tl i);
             assert (index l 0 = a);
             assert (count a l = 1 + count a tl);
             counting_lemma tl a

Second, we prove that if two boolean lists l1 and l2 of equal length are such that for every true element of l1 there is a true element of l2 at the same index, then l2 has at least as many true values as l1. To get this proof to work, we need to make some observations about the relationship between indices in a list and in the tail of the list:

val increasing_count_lemma: l1: list bool -> l2: list bool{length l1 = length l2} ->
              Lemma (requires forall j. (index l1 j = true) ==> (index l2 j = true))
                    (ensures count true l2 >= count true l1)
let rec increasing_count_lemma l1 l2 = match l1, l2 with
  | h1 :: tl1, h2 :: tl2 -> assert (h1 = index l1 0);
                         assert (forall k. (index tl1 k = index l1 (k + 1)));
                         assert (forall k. (index tl1 k = true) ==> (index tl2 k = true));
                         increasing_count_lemma tl1 tl2;
                         assert (count true l2 >= count true l1)
  | [], [] -> ()

Third, we prove that if two lists of equal length agree in each index, then the lists are equal:

val pointwise_list_equality_lemma: l1: list bool -> l2: list bool{length l1 = length l2} ->
              Lemma (requires forall i. (index l1 i = index l2 i))
                    (ensures l1 = l2)
let rec pointwise_list_equality_lemma l1 l2 = match l1, l2 with
  | [], [] -> ()
  | h1 :: tl1, h2 :: tl2 ->
    assert (index l1 0 = h1 /\ index l2 0 = h2);
    assert (h1 = h2);

    assert (forall j. (index tl1 j = index l2 (j+1)));
    pointwise_list_equality_lemma tl1 tl2;
    assert (tl1 = tl2)

Finally, we prove that if two boolean lists agree in every index except one, then the number of true elements cannot differ by more than 1 between the lists.

val decreasing_count_lemma: l1: list bool -> l2: list bool{length l1 = length l2} -> i:nat{i < length l1} ->
                  Lemma (requires forall j. (i = j \/ index l1 j = index l2 j))
                        (ensures count true l2 >= (count true l1) - 1)
let rec decreasing_count_lemma l1 l2 i = match l1, l2 with
  | h1 :: tl1, h2 :: tl2 ->
    assert (forall j. (index tl1 j = index l1 (j+1)));
    match i with
      | 0 ->
        assert (forall j. (index tl1 j = index tl2 j));
        pointwise_list_equality_lemma tl1 tl2;
        assert (tl1 = tl2);
        assert (count true tl1 = count true tl2)
      | _ ->
        assert (h1 = index l1 0 /\ h2 = index l2 0);
        assert (h1 = h2);
        decreasing_count_lemma tl1 tl2 (i-1)

With these lemmas in place, we are ready to return to the water tank example.

Proving properties of stateful functions using ghost values

Let us now consider the overflow constraint. It says that if the input valve, i.e. valve 0, is open then at least two of the other valves must also be open. We begin by writing a pure function non_overflowing which determines if this property holds for some list:

val non_overflowing: state -> bool
let non_overflowing vs = match vs with
  vi :: vos -> (not vi) || count true vos >= 2

This function splits the state into the state of the input valve, vi, and the states of the output valves, vos. A state in which the water tank does not overflow is one where either the input valve is closed, i.e. not vi holds, or where the number of open output valves, count true vos, is at least two.

Our first goal is to prove that the result of calling open_all' is a state where the tank is not overflowing. The reason this holds is that if every valve is open, then the number of open output valves is equal to the total number of output valves, which is greater than or equal to the 2 required for not overflowing. This can be expressed in a pure lemma, using the counting_lemma from the previous section, as follows.

val all_open_is_non_overflowing: vs: state ->
                 Lemma (requires (forall i. (index vs i) = true))
                       (ensures non_overflowing vs)
let all_open_is_non_overflowing vs = match vs with
  vi :: vos ->
  assert (forall (i:nat{i+1 < length vs}). index vs (i+1) = index vos i);
  counting_lemma vos true;
  assert (count true vos = length vos);
  assert (non_overflowing vs)

We now need to prove that this lemma translates to a property of the stateful function open_all'. The crucial part in being able to do this is having access to the state of the water tank system in function definitions, in order to invoke the all_open_is_non_overflowing lemma on the final state of open_all'. The type signature of a function reading the state is simple. It would look something like this: unit -> Tank state (fun vs0 -> True) (fun vs0 x vs2 -> vs0 = x /\ vs0 = vs2). However, we have no function in the water tank interface which allows us to read the state, so we cannot supply any implementation of this function. Since we only intend to use it in type signatures and proofs, which are erased when the code is extracted, this is not a problem in itself. However, having a function which crashes or behaves incorrectly at runtime if you use it in the wrong context is dangerous. Luckily, there is a way of resolving this using FStar.Ghost.

Before getting to FStar.Ghost, let us briefly discuss the effect GTot. Functions with the GTot effect are total functions which can be used freely in type specifications, but not at all in function bodies. They represent functions which cannot actually be executed at runtime, but can be reasoned about at type checking time. As an example, consider the function f defined as follows.

val f: nat -> GTot nat
let f n = n + 5

As promised, this function can be used to describe the types of other functions, so this definition of g type checks:

val g: n:nat{f n < 10} -> k:nat{f k < 12}
let g n = n + 2

However, trying to type check the following definition of h gives the error Computed type "nat" and effect "GTot" is not compatible with the annotated type "nat" effect "Tot".

val h: nat -> nat
let h n = f (n + 2)

The module FStar.Ghost contains two important functions called hide and reveal, and a type called erased. Their types and implementations are as follows:

[@erasable]
noeq
type erased (a:Type) =
  | E of a

val hide: #a:Type u#a -> a -> Tot (erased a)
let hide #a x = E x

val reveal : #a:Type u#a -> erased a -> GTot a
let reveal #a (E x) = x

The hide function takes any value, and wraps it in erased, which in itself is not very exciting. However, the only way of getting the value back is by calling reveal to unwrap it again. Since reveal has effect GTot, any value which has been hidden by hide can only be used in type specifications.

We can use this fact to define a get_tank_state function, which gives the current state of the water tank system. The fact that get_tank_state () is the current state of the system is encoded in the post-condition vs0 = reveal x. Since the return value of get_tank_state has type erased state, which the compiler extracts to the type unit, it cannot actually be used at runtime, so we do not need to specify an implementation of the function.

assume val get_tank_state: unit -> Tank (FStar.Ghost.erased state)
                                (fun vs0 -> True)
                                (fun vs0 x vs1 -> vs0 = FStar.Ghost.reveal x /\ vs0 = vs1)

With this, we can define a function open_all which does exactly what open_all' does, but additionally ensures that the resulting state of the water tank system is not overflowing. The only difference in the types of open_all' and open_all is that we have now added the post-condition non_overflowing vs1. We define open_all by first calling open_all', and then capturing the resulting state with get_tank_state in order to pass it to the lemma all_open_is_nonoverflowing. This is sufficient for the stronger post-condition to be accepted by the type checker.

val open_all: unit -> Tank unit
                          (fun _ -> True)
                          (fun _ _ vs1 -> (non_overflowing vs1) /\
                                       (forall i. (index vs1 i) = true))
let open_all () = open_all' ();
  let vs = get_tank_state () in (
    all_open_is_non_overflowing vs;
    assert (non_overflowing vs))

Completing the safe water tank controller interface

With the definition of open_all in place, we can proceed similarly to give definitions of the other functions. The definition of close_all is completely analogous to open_all: We add the post-condition non_overflowing vs1, and use get_tank_state to be able to reason about the resulting state. In this case, to see that the tank is not overflowing it is sufficient to observe that the input valve is closed.

val close_all: unit -> Tank unit
                           (fun _ -> True)
                           (fun _ _ vs1 -> (non_overflowing vs1) /\
                                        (forall i. (index vs1 i) = false))
let close_all () = close_all' ();
  let vs = get_tank_state () in (
    assert (index vs 0 = false)
  )

The definition of open_valve is more interesting, since it cannot be called at any time without causing the water tank to overflow. In other words, it requires additional preconditions to ensure that the resulting state is non-overflowing. The necessary preconditions, and a proof that they are sufficient, are as follows:

val open_valve: i: valve -> Tank unit
            (fun vs -> non_overflowing vs
                  /\ index vs i = false
                  /\ (i > 0 \/ count true (tail vs) >= 2))
            (fun vs0 _ vs1 -> non_overflowing vs1
                  /\ index vs1 i = true
                  /\ (forall j. i = j \/ index vs0 j = index vs1 j))
let open_valve i =
  let vs1 = get_tank_state () in
  open_valve' i;
  let vs2 = get_tank_state () in
  assert (i > 0 /\ index vs1 0 = false ==> index vs2 0 = false /\ non_overflowing vs2);

  assert (forall j. ((index vs1 j = true) ==> (index vs2 j = true)));
  assert (forall j. index (tail vs1) j = index vs1 (j+1));
  assert (forall j. ((index (tail vs1) j = true) ==> (index (tail vs2) j = true)));
  increasing_count_lemma (tail vs1) (tail vs2)

The only function which remains before we have a safe interface to the water tank controller is close_valve. It’s definition is similar to the one for open_valve:

val close_valve: i: valve -> Tank unit
              (fun vs -> non_overflowing vs
                    /\ index vs i = true
                    /\ (i = 0 \/ count true (tail vs) >= 3))
              (fun vs0 _ vs1 -> non_overflowing vs1
                           /\ index vs1 i = false
                           /\ (forall j. i = j \/ index vs0 j = index vs1 j))
let close_valve i =
  let vs1 = get_tank_state () in
  close_valve' i;
  let vs2 = get_tank_state () in
  match i with
    | 0 -> assert (non_overflowing vs1)
    | _ ->
      assert (forall j. (index (tail vs1) j = index vs1 (j+1)));
      decreasing_count_lemma (tail vs1) (tail vs2) (i-1);
      assert (1 + count true (tail vs2) = count true (tail vs1))

Conclusion

In this post, we started from a hypothetical external library containing functions for manipulating the state of an external system. Using the effect system in F*, we created a wrapper for the library which enforces certain correctness conditions at compile time. We did so by defining a custom state effect using STATE_h. The implementation was conceptually split into two parts. The first part was to give the functions in the library types which faithfully model the effect they have on the external state, but which do not prevent the functions from being used incorrectly. The second part was to refine the types by adding preconditions which allow the type checker to verify that the functions are not called when it is not safe to do so.

Appendix

Predicate transformers, weakest preconditions, and effect lifting

This appendix explains the definition of the function pure_lift from the section “Defining an effect”. The explanation here is intended to be true in spirit, but not in detail; it has been simplified, and is not true if taken literally. In particular, the types we give for predicates here have codomain bool, rather than the more correct Type.

This topic is also explained in the section “9. Specifying effects” (in particular in sections “9.1. Background: Generating compact verification conditions”, “9.3. The PURE effect”, and “9.5. Lifting effects”) in the F* tutorial. However, that section is not easy to absorb on a first read-through, so I would recommend reading, in parallel with Section 9 of the tutorial, the following papers which, while significantly longer and deeper, provide more context:

  • Dijkstra Monads for Free (Danel Ahman, Catalin Hritcu, Kenji Maillard, Guido Martínez, Gordon Plotkin, Jonathan Protzenko, Aseem Rastogi, Nikhil Swamy), In 44th ACM SIGPLAN Symposium on Principles of Programming Languages (POPL), ACM, 2017.
  • Verifying Higher-order Programs with the Dijkstra Monad (Nikhil Swamy, Joel Weinberger, Cole Schlesinger, Juan Chen, Benjamin Livshits), In Proceedings of the 34th annual ACM SIGPLAN conference on Programming Language Design and Implementation, 2013.

Suppose that we have some function f: b -> Tot a, and some property p: a -> bool, and that we want to prove that the return value of f always satisfies the property p. What do we need to prove to ensure that this is the case? What we need to prove depends on the function f; if f is a constant function which always return the same value y, then we only need to prove that p y is true, but in general we would need to prove something more involved. When we define the function f, F* automatically computes something called a “predicate transformer” corresponding to f. This predicate transformer, which we will denote wp, takes any predicate a -> bool, for instance p, and computes what needs to be proved to ensure that the return value of f always satisfies p. All such predicate transformers for pure functions have the type pure_wp, which is defined in ulib/prims.fst in the F* standard library.

Consider now the analog of this for functions with the TANK effect, which can modify some state of type state. The properties we would like to prove about the return type of some TANK function have the form p: a -> state -> bool rather than just p: a -> bool as for pure functions, since we are interested in proving things about both the return value (which has type a), and the resulting state after the function has been executed (which has type state). In addition, what needs to be proved to ensure that some property p holds for the resulting state and return value depends on the state before the function has been run. Taking all this together, the analog of wp (corresponding to the pure function f above) for a function g with the TANK effect would be something which takes as arguments a property p: a -> state -> bool and an initial state x: state, and computes what needs to be proved in order for the property p to hold for the resulting state and return value of g.

We are now in a position to explain what pure_lift is doing in the section Defining an effect. It takes as an argument wp, which is the predicate transformer corresponding to some pure function f. It then produces a predicate transformer for the same function f, but viewed as a TANK function rather than a PURE function. This new predicate transformer takes a property p: a -> state -> bool, which we would like to prove holds for the resulting state and return value of f, together with the initial state x: state which is the state before running f. To prove that the property p holds after the function f has been executed, what do we need to prove? If we denote the return value of f by y, and the resulting state by x1, then what we want to prove is p y x1. However, f is a pure function, so x1 = x. This means that we want to prove p y x, where y is the return value of f. Hence the property we want to prove after observing that x1 = x is fun y -> p y x, which has the type a -> bool. This is now a property which does not depend on the resulting state of f, and hence can be considered a property of the pure function f (albeit parameterized by an initial state x). At our disposal we have wp, which tells us what needs to be proved to ensure that some property holds for f viewed as a pure function. We can plug the property fun y -> p y x into wp, and that will tell us what needs to be proved in order to guarantee that p holds. We have then arrived at the definition of pure_lift:

unfold let pure_lift (a:Type) (wp:pure_wp a) = fun (p: a -> state -> Type) (x: state) -> wp (fun y -> p y x)
